A privacy-first, local-first calendar for Apple platforms. Sync over Tailscale only — peer mesh, 1–3 devices. No cloud. No server. No backend.

01 — Product

What Meridian Is

A calendar grounded in Lisa Feldman Barrett's constructed emotion theory. The affect layer exists not to track mood but to surface schedule patterns — using Euclidean prediction error in three-dimensional affect space as a signal for when a time window is running differently than the user's own history predicts.

The system's only response to detected pattern deviation is a schedule-review prompt. It never names an emotion. It never suggests a cause. It never fires from a single data point.

stack
SwiftUI throughout
iPhone · iPad · macOS menu bar. No Electron, no React Native, no cross-platform bridges. v1 ships revenue-generating Apple product before any Rust translation layer.
sync
Tailscale only
Peer mesh transport. Cloud adjacency is not a tradeoff — it is a category error for this product. Any architecture introducing cloud dependency is a different product.
CRDT
Automerge-Swift
Pure Swift, ARM-native, no FFI. Correct offline-first convergence without a linearization point. closed D2-01

03 — Identity

Five named defaults. Labels are user-editable strings — not enums. Internal identifier is a stable UUID. Profile behavior travels with the UUID, not the label. A rename does not produce a merge conflict.

Events may be assigned to multiple profiles simultaneously. No profile is "primary." No profile implies it is for "serious" users and others for casual ones.

04 — Theory

Barrett Constructed Emotion

Lisa Feldman Barrett's active inference account (PMC5390700) establishes the constraints this system operates under. The brain generates top-down predictions about internal body state. Conscious affect is the brain's best guess, updated by incoming signals.

Critical implication: Meridian's prediction error signal is not a proxy for "the user is stressed." It is a proxy for "this time window is producing more physiological uncertainty than the user's own history predicts it should." The response is a schedule observation — not a wellness intervention.

What the system must never do

never
Name an emotion category. Emotion words are user-constructed. The system has no access to them.
never
Display valence or arousal as a self-description. Showing "your average valence is −0.4" produces iatrogenic mood effects.
never
Suggest a cause for prediction error. The system sees dimensions and schedule data. Causal claims are confabulation.
never
Fire a response from a single high-error check-in, regardless of ε magnitude. One point is noise.
never
Surface prediction error magnitude. ε is internal. Showing it creates gamification dynamics that distort future check-in behavior.

07 — Signal

Prediction Error

Euclidean distance in 3D affect space ε = √( (Vnew − µ_V)² + (Anew − µ_A)² + (Cnew − µ_C)² )
max ε
√(4 + 4 + 1) = 3.0 — maximum possible distance across all three dimensions at their extremes.
threshold
ε_threshold = 0.85 — ~28% of maximum distance. Approximately 2–3 combined standard deviations. D1-05
trigger
N = 3 high-error check-ins in the same bucket within a rolling 7-day window. Not calendar week — rolling from most recent check-in timestamp. D1-06
cooldown
14 days per bucket after prompt shown or dismissed. No further prompts for that bucket during cooldown. D1-07

09 — Sync

CRDT Selection closed D2-01

selected
Automerge-Swift
Pure Swift, ARM-native via Xcode toolchain. No FFI. Correct offline-first convergence. Only candidate satisfying all three hard constraints simultaneously.
out
Y.js / Yrs via C FFI
Rust FFI bridge. Second toolchain. Invisible to LLDB. Breaks on Xcode updates. 40–60% memory advantage not meaningful at Meridian's v1 workload size.
out
SQLite + LWW
Structurally broken under clock skew in a peer mesh without NTP-synchronized server. Silent destructive overwrites. No history. No recovery.
out
Custom OT
Requires a linearization point — a server or consensus node. Incompatible with Tailscale-only architecture. Eliminated immediately.
10 — Schema

Data Schema

// EventDoc — top-level CRDT document
struct EventDoc: Codable {
    var schemaVersion: Int
    var events: [String: EventRecord]            // keyed by Meridian UUID
    var contextProfileMappings: [String: [String]] // eventID → [profileIDs]
    var softHolds: [String: SoftHoldRecord]
    var buffers: [String: BufferRecord]
}

// AffectDoc — separate CRDT document
struct AffectDoc: Codable {
    var schemaVersion: Int
    var checkIns: [String: AffectCheckIn]        // keyed by UUID
    var bucketBaselines: [String: BucketBaseline]
    var promptLog: [String: PromptLogEntry]
}

// AffectCheckIn
struct AffectCheckIn: Codable {
    var id: String
    var valence: Float        // −1.0 to +1.0
    var arousal: Float        // −1.0 to +1.0
    var clarity: Float        //  0.0 to +1.0
    var bucketID: String
    var eventRef: String?     // optional link to EventRecord.meridianID
    var submittedAt: Date
    var epsilon: Float?       // nil if bucket is FORMING — never surfaced to user
    var isHighError: Bool
}

Merge semantics

title, loc, notes
Register — deterministic winner, losing value retained in history.
startDate, endDate
Register — later actor wins. Both values in history.
profileMappings
Set — additive merge. Both offline additions survive. Correct behavior.
checkIns
Map — concurrent additions merge cleanly. Unique UUIDs; no conflicts possible.
11 — Transport

Sync Protocol

GET  /meridian/sync/status
     → { peerID, lastSyncAt, eventDocCursor, affectDocCursor }

POST /meridian/sync/event-doc
     Authorization: MeridianAuth [shared secret from Keychain]
     body: Automerge binary delta since cursor

POST /meridian/sync/affect-doc
     Authorization: MeridianAuth [shared secret from Keychain]
     body: Automerge binary delta since cursor
port
47301 — user-configurable in Settings. Bound to Tailscale IP only. Does not bind to non-Tailscale interfaces.
auth
Shared secret in Keychain. Exchanged at device onboarding. Not hardcoded. Not in UserDefaults.
triggers
App foreground (last sync > 5 min) · explicit tap · post-mutation debounced 30s.
delta size
< 50 KB per sync pair for typical daily usage (5–10 event mutations, 2–3 check-ins).
12 — QA

Cultural Competence

Cultural competence = the system does not require users to distort, hide, simplify, or perform their lives differently in order to use the calendar effectively. Not a values statement. Each property below is pass/fail from code audit.

P1 Role Non-Exclusivity

A single event is assignable to multiple context profiles. No language or UI implies one assignment is "correct" and others supplemental. "Other" as a catch-all is a fail condition.

P2 Profile Label Non-Othering

The five defaults do not form an implicit hierarchy from "serious" to "casual." A mutual aid coordinator should not be forced between "Ops" (corporate framing) and "Minimal" (implies low-structure work).

P3 Enrichment Non-Normalization

Affect-pairing logic does not position any cultural, spiritual, or productivity paradigm as the correct response to a given state. ≥30% of each content pool is culturally neutral. No opt-in gate for non-Western content.

P4 Temporal Home Screen Universal Readability

Three time states (7am, 2pm, 10pm) work for non-conventional schedules, non-single-household contexts, and non-neurotypical morning relationships. "Workday," "your day," "tonight" in assumed contexts are fail signals.

P5 Affect Check-In Non-Prescriptiveness

The check-in UI does not imply, through color, visual weight, or spatial metaphor, that any affect position is more desirable than another. High-valence/high-arousal corner rendered warmer or more saturated is a fail condition.
